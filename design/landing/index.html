What shunt does differently

KV Cache Reuse

Routes requests to the model instance that already has the longest matching KV cache prefix. Fewer tokens computed means lower cost and faster responses.

OpenAI-Compatible

Drop-in replacement for the OpenAI API. Point your existing SDK or client at shunt and it just works. No code changes. No vendor lock-in.

Built in Zig

Compiled, fast, and memory-safe. No runtime, no GC pauses. shunt handles thousands of concurrent streaming connections with predictable latency.

Real-Time Metrics

Cache hit rates, cost savings, latency percentiles — all exposed via a dashboard and API. See exactly how much you are saving, in real time.

Cost Reduction

Average 60-90% cost savings on prompt token processing. If the cache already has your prefix, you do not pay to recompute it.

AGPL v3

Fully open source. Inspect the code, fork it, contribute. No enterprise tier holding back core features. The load balancer is free. The savings are yours.

How it works

1

Point your client at shunt

Replace api.openai.com with your shunt endpoint. Same SDK, same request format. Nothing else changes.

2

shunt checks the cache

For every incoming request, shunt scans your model instances and finds the one with the longest matching KV cache prefix.

3

You ride the cache

The request goes to the instance that already computed your prefix. Cached tokens are free. You pay only for what is new.

Get started in 30 seconds

$ curl -sSL https://get.shunt.dev | sh
$ shunt --listen :8080 \
--backend gpt-4o:api-key-1 \
--backend gpt-4o:api-key-2 \
--backend gpt-4o:api-key-3
✓ shunt listening on :8080
✓ 3 backends registered for gpt-4o
✓ Dashboard at http://localhost:8080/dashboard

That is it. Point your OpenAI SDK at http://localhost:8080/v1 and shunt handles the rest.

Prefer Docker? docker run -p 8080:8080 ghcr.io/shitware-industries/shunt:latest
